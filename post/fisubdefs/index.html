Docker 镜像制作(实操)

2021-12-10 / 41 min read

Docker镜像有没有内核？
从镜像大小上面来说，一个比较小的镜像只有十几MB，而内核文件需要一百多兆， 因此镜像里面是没有内核的，镜像在被启动为容器后将直接使用宿主机的内核，而镜像本身则只提供相应的rootfs，即系统正常运行所必须的用户空间的文件系统，比如/dev/，/proc，/bin，/etc 等目录，所以容器当中基本是没有/boot目录的，而/boot当中保存的就是与内核相关的文件和目录。

为什么没有内核？ 
由于容器启动和运行过程中是直接使用了宿主机的内核，所以没有直接调用过物理硬件，
所以也不会涉及到硬件驱动，因此也用不上内核和驱动，另外有内核的那是虚拟机。

6.1：手动制作yum版nginx镜像： (了解即可)

Docker image制作类似于虚拟机的镜像制作，即按照公司的实际业务务求将需要安装的软件、相关配置等
基础环境配置完成，然后将其做成镜像，最后再批量从镜像批量生产实例，这样可以极大的简化相同环境的部署工作，
Docker 的镜像制作分为手动制作和自动制作(基于 DockerFile)，企业通常都是基于 Dockerfile 制作镜像，
其中手动制作镜像步骤具体如下： 

1> 下载一个基础的官方镜像(centos ubuntu)
    docker pull centos

2> 基于基础镜像启动一个容器,并进入到容器
    docker run -it -p 8801:80 centos bash

3> 在容器里面做配置操作
    安装基础命令 配置运行环境 安装服务和配置服务 放代码
      yum install epel-release -y  安装yum源
      yum install nginx -y
      rm -rf  /usr/share/nginx/html/index.html
      echo "linux38" >>  /usr/share/nginx/html/index.html

4> 提交为一个新镜像
    重新开启一个窗口
    docker commit [选项] container [镜像名称:tag]
       -a 指定镜像的作者信息
       -c 添加指令,传递dockerfile指令
       -m 指定一个message  提交信息(备注)
       -p 暂停,做镜像的时候会暂停容器的使用(默认操作)
    
    docker commit -m "nginx v1"  068891201bfd   linux38:nginx
    相当于将容器复制成一个镜像
    
5> 基于自己的的镜像创建容器并测试访问
   docker run -it -p 80:80 linux38:nginx bash 
   docker run -it -d -p 80:80 linux38:nginx  nginx -g "daemon off"
   nginx 需要将终端功能后端关闭,提到前台,让nginx在容器中的前端去执行
   否则会因为没有PID为1的进程,无法将容器支撑长久,就会挂掉
   更改nginx配置文件,添加 daemon off;

6.2：DockerFile制作编译版nginx 1.16.1 镜像

DockerFile可以说是一种可以被Docker程序解释的脚本，DockerFile是由一条条的命令组成的，
每条命令对应linux下面的一条命令，Docker程序将这些DockerFile指令再翻译成真正的linux命令，
其有自己的书写方式和支持的命令，Docker程序读取DockerFile并根据指令生成Docker镜像，
相比手动制作镜像的方式，DockerFile更能直观的展示镜像是怎么产生的，
有了写好的各种各样DockerFile文件，当后期某个镜像有额外的需求时，
只要在之前的 DockerFile添加或者修改相应的操作即可重新生成新的Docke镜像，
避免了重复手动制作镜像的麻烦，具体如下： 
https://docs.docker.com/engine/reference/builder/    

Dockerfile添加内容指令:
ADD      从当前目录下往镜像中添加配置文件、压缩包或者目录,当遇到tar.gz的时候将会自动解压
COPY    功能和ADD一样,但是不会解压tar.gz包
ENV      给容器设置环境变量使用,通过环境变量的方式传到容器中,
        当容器中需要通过变量获取账号密码的时候可以使用,容器中的任何一个地方都可以取到这些值
EXPOSE  定义要暴露的端口
FROM    指定父镜像
LABEL   定义镜像的属性信息
STOPSIGNAL  发送信号的指令  (不常用)
USER
VOLUME
WORKDIR 
RUN   直接执行shell命令

Dockerfile 参考: 
Docker可以通过阅读Docker的指令来自动构建镜像 Dockerfile。
A Dockerfile是一个文本文档，其中包含用户可以在命令行上调用以组装图像的所有命令。
使用docker build 用户可以创建自动构建，该构建连续执行多个命令行指令(shell命令)
可以在多个Dockerfile中写多行,每行可以写多个指令 ,指令其实就是shell命令
对于我们来说,就是在dockerfile中写入原生的shell命令,
docker在执行镜像构建的过程当中,指令是从上到下逐行执行的

用法
该docker build命令根据Dockerfile和上下文构建镜像。
构建的上下文是指定位置PATH或的文件集URL。
这PATH是本地文件系统上的目录。该URL是一个Git仓库的位置。
但是一般情况下会将dockerfile放在本地

上下文是递归处理的。
因此，a PATH包括任何子目录，并且a PATH 包括URL存储库及其子模块。
此示例显示了一个使用当前目录作为上下文的构建命令：

$ docker build .    直接在当前目录下执行,
通常情况下,会进入到宿主机的某个目录,在目录中放入一个dockerfile,然后执行docker build .
Sending build context to Docker daemon  6.51 MB

如果不在本地的话需要适应-f选项指定路径
 docker build -f /path/to/a/Dockerfile . 
 
如果构建成功，则可以指定一个存储库和标记，用于在其中存储新图像：
$ docker build -t shykes/myapp .

要在构建后将映像标记到多个存储库中(存放镜像的地方)，请在-t运行build命令时添加多个参数：
$ docker build -t shykes/myapp:1.0.2 -t shykes/myapp:latest .

在Docker守护程序运行中的指令之前Dockerfile，它会对进行初步验证，
Dockerfile如果语法不正确，则返回错误：

$ docker build -t test/myapp .
Sending build context to Docker daemon 2.048 kB
Error response from daemon: Unknown instruction: RUNCMD

dockerfile 层级结构

JAVA环境结构:     
第一层: 官方的基础镜像 Ubuntu/centos/debain
第二层: 公司的基础镜像 (baseimage) 在官方镜像的基础上,安装常用命令,不同的系统打出不同的镜像
第三层：根据公司业务在baseimage的基础上,打不同版本的JDK镜像,给原生的java服务使用
第四层: 在java的基础上,打不同版本的tomcat镜像
第五层: 打不同版本的app应用, 
这样就可以上传到公司内部的镜像仓库(harbor)上了

运行容器的宿主机 各种node节点,要保证自动的去公司内部的镜像仓库(Harbor)中拉取镜像
各个节点拉取到镜像以后就可以运行容器

NGINX环境结构: 
在公司基础镜像的基础上 
第三层: 安装不同版本nginx或者PHP,制作镜像
第四层: 静态服务,并添加前端代码
        在第三层的基础上打静态服务镜像,将代码、配置文件放到静态服务镜像中,就可以启动服务
        将镜像上传到公司内部的镜像仓库(Harbor)中,各个node节点就可以在Harbor上拉取镜像,对外提供服务

如果有公共数据的话例如: 图片、静态文件等较大文件,将会放到存储(NAS)中
NAS的数据是给容器使用的,但是NAS挂载在node节点上,然后再给各个容器去使用,

容器一定是可以直接或者间接的去使用到存储中的数据,而且每个容器都可以是使用,从而保证数据的统一性
数据是由java容器往里写,有nginx服务器往外读,javar容器间接的或者直接的将数据写进NAS存储中
通过node节点,访问存储,往存储中写入数据,然后nginx就可以读取NFS中的数据
所以他们实际上看到的是同一份数据,
在镜像分层上，不会直接从原始的官方的镜像直接打到公司的业务镜像,因为中间到操作的步骤太多
会将多个步骤拆分开,每一层都是一个镜像

profile文件是不生效的,对于容器来说,不是通过ssh也不是通过ssh登录的,

    1/下载一个基础的官方镜像(centos ubuntu)
        docker pull centos
    2/基于基础镜像启动一个容器,并进入到容器
         docker run -it -p 8801:80 centos:7.6.1810 bash
    3/在容器里面做配置操作
        安装基础命令 配置运行环境 安装服务和配置服务 放代码
        #yum install epel-release -y
        #yum install nginx -y
        ## rm -rf  /usr/share/nginx/html/index.html
        # echo "linux38" >>  /usr/share/nginx/html/index.html
    4/提交为一个新镜像
        docker commit
    5/基于自己的的镜像创建容器并测试访问

基于Dockerfile制作镜像: 学习docker最重要的地方
    1/下载官方基础镜像(ubuntu/alpine/centos)
    2/先构建出安装了基础命令和创建常用账户的企业基础镜像/改时区
    3/构建运行环境的镜像(装JDK),方便后期多个java业务调用当前的不同版本的JDK镜像
    4/构建不同版本的tomcat/nginx镜像,方便后期不同业务或者组件调用
    5/基于上次的包含运行环境的镜像构建出业务镜像(放java代码/前端页面)
