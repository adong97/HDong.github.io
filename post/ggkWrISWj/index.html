一、传统架构代码升级

传统情况下，在虚拟机进行代码升级与回滚。公司内部采用 gitlab ，由开发将代码提交，通过Jenkins进行代码部署：

  1. 先执行代码 clone
  2. 代码 clone 之后就会通过 maven 进行编译
  3. 将编译好的包拷贝至虚拟机的（scp、ansible）一个新目录
  4. 拷贝过去之后如果需要对代码进行替换，切换软连接重启服务（将虚拟机下线，主要是担心用户的访问会转发到虚拟机上，从而导致虚拟机报错）
  5. 停掉对应服务
  6. 下线之后再将代码换掉，替换代码有两种方法：
    • 将之前存储代码的目录清空，或者将之前存储代码的目录全删完然后在将目录替换
    • 或者通过软连接的方式进行替换
  7. 启动服务，其实这时候就会使用的我们上一步加载的新代码
  8. 测试，因为服务起来之后为了确保能够正常访问所以需要提前测试（通过 curl 命令或者简单的 API 进行测试）
  9. 测试通过上线
  10. 然后在负载均衡中将需要升级的主机从负载均衡里面摘掉，然后升级完成在加回负载均衡，并做灰度发布，这样的话我在升级过程中是不影响用户访问的。如果在以此类推升级其他服务器
  11. 然后灰度发布，如上图有两个版本是 v1 有一个本版是新发布的 v2 ，即使有用户访问有问题也只能说影响那百分之 30 左右的用户，如果这时候 v2 版本 有问题就直接回滚为 v1 ，如果 v2 没有问题，我们直接将其他版本升级到 v2
  12. 没有问题之后部署到负载均衡
  13. 最后用户就能直接访问

二、K8S 架构代码升级

执行滚动更新:
https://kubernetes.io/zh/docs/tutorials/kubernetes-basics/update/update-intro/

用户希望应用程序始终可用，而开发人员则需要每天多次部署它们的新版本。在 Kubernetes 中，这些是通过滚动更新（Rolling Updates）完成的。 滚动更新允许通过使用新的实例逐步更新 Pod 实例，零停机进行 Deployment 更新。新的 Pod 将在具有可用资源的节点上进行调度。

在前面的模块中，我们将应用程序扩展为运行多个实例。这是在不影响应用程序可用性的情况下执行更新的要求。默认情况下，更新期间不可用的 pod 的最大值和可以创建的新 pod 数都是 1。这两个选项都可以配置为（pod）数字或百分比。 在 Kubernetes 中，更新是经过版本控制的，任何 Deployment 更新都可以恢复到以前的（稳定）版本。

在 K8S 中进行代码升级相对来说就比较简单，K8S 也是通过滚动升级来实现代码更新的，一般都是 3 个 pod 跑一个业务。

在升级过程中 svc 会把请求逐渐的转向新的。升级的话（v1 到 v2），K8S 中会先启动一个新的Pod将其升级为 V2 版本，等 v2 版本起来之后将三个 pod 中的其中某一个删除，并且 svc 会将删除掉的 pod 的请求往 v2 pod 进行转发。

当这个删除掉老的 v1 之后会在启动一个 v2 的 pod ，并且这时在将一个 v1 版本的删除掉，然后在启 v2 pod。

所以通过这种方式来保证我们在代码升级中是可用的，不会说将 pod 全部删除而导致用户无法访问，从而实现了零停机进行控制器的版本更新。

滚动更新允许以下操作：

  • 将应用程序从一个环境提升到另一个环境（通过容器镜像更新）
  • 回滚到以前的版本
  • 持续集成和持续交付应用程序，无需停机

2.1 K8s镜像更新方式简介及示例

常见的更新方式有三大类:

  1. 更新yaml文件中image的版本，然后kubectl apply -f app.yaml
  2. 使用kubectl命令实现镜像更新及回滚。kubecetl set image
镜像更新命令格式为： 
$ kubectl set image deployment/deployment-name container-name=image -n namespace 
$ kubectl set image deployment/magedu-tomcat-app1-deployment magedu-tomcat-app1-container=registry.cn-hangzhou.aliyuncs.com/tomcat-app1:v2 -n magedu 
deployment.apps/magedu-tomcat-app1-deployment image updated

查看历史版本信息： 
$ kubectl rollout history deployment/magedu-tomcat-app1-deployment -n magedu deployment.apps/magedu-tomcat-app1-deployment 
REVISION CHANGE-CAUSE 
3               <none> 
4               <none>

回退到上一个版本： (只能在最近两个版本进行回退)
$ kubectl rollout undo deployment/magedu-tomcat-app1-deployment -n magedu 
deployment.apps/magedu-tomcat-app1-deployment rolled back

回退到指定版本： 
$ kubectl rollout undo deployment/magedu-tomcat-app1-deployment --to-revision=4 -n magedu
deployment.apps/magedu-tomcat-app1-deployment rolled back

https://kubernetes.io/zh/docs/concepts/workloads/controllers/deployment/#pausing-and-resuming-a-deployment

$ kubectl rollout pause deployment magedu-tomcat-app1-deployment -n magedu

2.3 镜像更新类型-Recreate简介

Recreate(重建更新)：
先删除现有的pod，然后基于新版本的镜像重建，Pod重建期间服务是不可用的，其优势是同时只有一个版本在线， 不会产生多版本在线问题，缺点是pod删除后到pod重建成功中间的时间会导致服务无法访问，因此较少使用。

Pod镜像更新过程中的状态：
具体流程为先删除旧版本Pod，然后deployment控制器再新建 一个ReplicaSet控制器并基于新版本镜像创建新的Pod，旧版本的ReplicaSet控制会保留用于版本回退。

kubectl describe replicasets magedu-tomcat-app1deployment-7bcb67c868 -n magedu
